# RoughCut Project Build Principles

**文档同步版本：** RoughCut v0.1.5（2026-04-27）

## 定位

这份文档定义 RoughCut 后续工程改动的默认构建原则。它优先级高于临时修补思路，用来约束实现方案、测试策略和自动化 agent 的执行方式。

## 核心原则

### 1. 主逻辑先成立

任何功能改动都必须先证明主生产路径合理、数据来源正确、职责边界清晰，再考虑异常拦截。

不能用失败用例去固化一个本身错误的生产方式。发现主路径在消费错误事实源、错误中间产物或错误职责边界时，优先修正主逻辑，而不是新增 guard、fallback 或失败测试把错误行为钉住。

### 2. 失败用例有准入门槛

失败用例只用于验证“合理主逻辑下的异常拦截”，包括缺输入、坏输入、外部服务失败、超时、重复执行、恢复和降级。

新增失败用例前必须先满足：

- 主路径 contract 已明确。
- 正常样例已经能产出正确结果。
- 异常分支不会替代主路径设计。
- 测试断言的是业务上应该拒绝的状态，而不是当前实现偶然产生的坏状态。

### 3. 回归测试锁正确行为

回归测试优先锁定正确的事实流、正确的输出 contract 和真实素材的可交付结果。历史失败样本可以进入回归集，但必须先完成根因修复，并把断言落在修复后的正确行为上。

### 4. Fallback 只能显式降级

fallback、guard、补偿和失败恢复只能作为显式降级路径存在，必须可观测、可定位、可重跑。不能把 fallback 设计成新的默认生产路径，也不能让它掩盖主链路职责错误。

### 5. 自动化 agent 必须遵守同一原则

Codex、ACP、Claude 或 Telegram agent 执行实现、构建、测试任务时，默认遵守本文件。尤其是涉及失败测试、异常拦截、fallback 和恢复逻辑时，必须先检查主链路是否合理，再决定是否增加失败用例。
