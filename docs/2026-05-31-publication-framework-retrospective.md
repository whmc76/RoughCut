# 2026-05-31 发布框架复盘

## 结论

这两天不是“没有发布能力”，而是“发布能力、严格核验、自动恢复”三层已经都有，但没有被收束成同一个状态机。

最核心的问题是：

- 发布前草稿态处理、发布中待回执、发布后终态核验，这三类状态被混在了一起。
- `submitted` 之后平台清空表单或跳转页面时，空字段快照被误判成草稿污染。
- 自动恢复随即触发 `clear_draft_context`，把“等待平台回执”的场景误导成“需要重发”的场景。

## 这次看到的真实症状

- `maxace 美杜莎4` 的 plan contract 是对的，`request_payload_plan_match=true`。
- 稳定平台多次停在 `submitted`，签名匹配，但 `response_payload` 字段快照为空。
- 系统把这种“提交后页面已清空”的现象当成 `content_plan_fill_gaps_pending / submitted_response_payload_empty_snapshot`。
- 随后自动恢复继续清稿、刷新、重跑，导致 live 试验很多，但没有闭环收口。

## 第一处变坏的边界

第一处变坏不在 packaging，也不在计划生成。

第一处变坏在：

- `real_release_gate` 对 `submitted` 态的解释。

具体表现：

- 计划签名是对的。
- 发布前字段校验是对的。
- 真实点击发布后，平台把表单清空或切到管理态。
- 系统却继续把“表单空了”当成“草稿脏了”。

这说明真正缺的是“发布后待回执/待终态核验”这一层状态，而不是更多草稿清理规则。

## 根因

根因可以拆成三点：

1. 状态机缺层
   没有把 `publish_receipt_pending` 作为独立状态。

2. 证据优先级错误
   `submitted` 后的空表单快照，被错误地高于“签名已匹配、请求已对账”的证据。

3. 恢复策略触发点过早
   自动恢复在还没证明“未发布”的情况下，就进入 `clear_draft_context`。

## 为什么这套框架还没真正落地

不是因为没有脚本，而是因为框架边界还没收住：

- 已经有 `preflight / release-gate / real-release-gate / autopilot / knowledge-base`。
- 但这些层更多是在累积验证与补丁，没有统一到一套稳定状态机。
- live 试验很多，异常也沉淀了不少，但“什么阶段允许清稿、什么阶段只允许核验”没有彻底固化。
- 所以框架看起来很全，实际上还在把不同状态当成同一种失败处理。

## 应该固化成的发布全流程

1. 草稿探测
   先确认目标平台当前 tab 是否已有旧草稿、旧素材、旧字段、错误路由。

2. 草稿决策
   只有明确存在旧草稿污染、失败残留或错误路由时，才允许清稿。

3. 发布前校验
   校验素材上传、字段回填、声明/合集/分类、计划签名、字段快照。

4. 真实发布
   执行最终发布/预约动作。

5. 发布后回执等待
   允许平台清空表单、跳转、进入审核/预约/管理态。

6. 发布后核验
   以最终链接、平台管理页、审核态、预约态为准，而不是只看提交后的表单快照。

7. 已发布收口
   只有公开链接或预约回执、签名、关键字段对账都成立，才判定通过。

8. 例外流程固化
   每个平台把真实遇到的异常分成：
   - 可直接恢复
   - 需要人工确认
   - 需要 LLM 探索根因后再升级框架

## 这次已落地的修正

- 在 `real_release_gate` 中新增了“发布后待回执”识别。
- 对 `submitted + signature_match` 的场景，不再默认建议 `clear_draft_context`。
- 自动恢复不再把这类场景直接视为“长时间卡死需要清稿”。
- runbook 已补充“先探测草稿、再决定清理、再发布、再做发布后核验”的顺序。

## 对 maxace 美杜莎4 的直接建议

下一轮不要再把 `submitted` 后的空表单当成草稿污染。

应按这个顺序推进：

1. 只跑稳定平台，优先 `douyin / xiaohongshu`。
